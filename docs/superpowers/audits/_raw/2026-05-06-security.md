### §A.1 SECRET_KEY 硬编码回退值 -- 生产 Session 可伪造
- **位置**: `app.py:68`
- **严重度**: 严重
- **根因诊断**: 代码逻辑 `_secret_key or "dev-change-me-in-production"` 在 `SECRET_KEY` 环境变量为空时静默回退到公开的硬编码字符串。判断条件 `FLASK_ENV != "development"` 只打印警告但不阻断启动。而项目 memory 记录生产环境运行在腾讯云 `124.221.23.173:5000`，一旦生产 `.env` 未设或丢失 `SECRET_KEY`，攻击者可用此已知字符串伪造 Flask session cookie，冒充任意用户（含 admin），实现完整账号接管。
- **修复建议**: 在 `FLASK_ENV != "development"` 且 `SECRET_KEY` 为空时调用 `sys.exit(1)` 强制中止启动，而非回退到默认值。
- **估时**: 0.5h
- **依赖**: 需确认生产 `.env` / Docker compose 已正确配置 `SECRET_KEY`；修改后 CI smoke test 需补一个 "无 SECRET_KEY 启动失败" 断言。

### §A.2 `/api/ai-engines` 未加 `@login_required` -- 信息泄露
- **位置**: `app.py:2946-2951`
- **严重度**: 低
- **根因诊断**: 该端点返回可用 AI 引擎列表（vendor / model / cost_hint），是唯一一个数据读取型 GET 端点缺少 `@login_required` 的。虽然不直接泄露 key，但暴露了后端引擎配置细节给未认证用户，便于攻击者做靶向信息收集。
- **修复建议**: 加 `@login_required`，与同级端点对齐。
- **估时**: 0.5h
- **依赖**: 无。

### §A.3 CSRF 豁免端点暴露在公网
- **位置**: `app.py:1410`, `app.py:1888`
- **严重度**: 中
- **根因诊断**: `@csrf.exempt` 加在 `/api/batch/<id>/start-mock` 和 `/api/single/_mock-task` 上，注释写 "dev-only"，但防护仅靠 `FLASK_ENV == "production"` 环境变量检查。若生产部署时未设置 `FLASK_ENV=production`（项目 `.env.example` 中并未包含此变量），CSRF 豁免即在公网生效，攻击者可构造跨站请求触发 mock 批处理流水线，消耗服务器资源。
- **修复建议**: (1) 删除 `@csrf.exempt`，mock 端点也应走 CSRF token；或 (2) 将 mock 端点注册到独立 Blueprint 并在生产环境彻底不注册（而非运行时 403）。同时在 `.env.example` 里补 `FLASK_ENV=production`。
- **估时**: 1h
- **依赖**: 需同步确认 Docker compose / 腾讯云部署脚本已设 `FLASK_ENV=production`。

### §A.4 登录无 rate-limit -- 暴力破解密码
- **位置**: `auth.py:13-36`
- **严重度**: 中
- **根因诊断**: `/auth/login` POST 端点无任何频率限制（全局未引入 `flask-limiter` 或等价组件）。攻击者可以无限次尝试用户名+密码组合。密码最低要求仅 6 字符（`auth.py:55`），弱密码在暴力攻击下秒破。5 个 demo 客户即将试用，公网可达，风险窗口已打开。
- **修复建议**: 引入 `flask-limiter`，对 `/auth/login` POST 设 `5/minute` per-IP 限制；同时考虑账号锁定策略（连续 N 次失败后锁 15 分钟）。
- **估时**: 2h
- **依赖**: 需 `pip install flask-limiter`；Redis 可选（默认 in-memory 够 demo 规模）。

### §A.5 Session Cookie 未设 Secure / SameSite 属性
- **位置**: `app.py` (全局 — 未设置 `SESSION_COOKIE_*` 配置)
- **严重度**: 中
- **根因诊断**: 生产环境目前跑在 `http://124.221.23.173:5000`（非 HTTPS），session cookie 和 remember_me cookie 均未设置 `Secure`、`SameSite` 属性。即使未来上 HTTPS，不显式配置 `SESSION_COOKIE_SECURE=True` 和 `SESSION_COOKIE_SAMESITE="Lax"` 也会留洞。当前 HTTP 环境下 cookie 可被中间人嗅探。
- **修复建议**: 立即加 `app.config["SESSION_COOKIE_HTTPONLY"] = True`（Flask 默认已是 True 但显式声明更安全）；`SESSION_COOKIE_SAMESITE = "Lax"`；HTTPS 上线后加 `SESSION_COOKIE_SECURE = True`。
- **估时**: 0.5h
- **依赖**: HTTPS 部署计划（Nginx 反代 + Let's Encrypt）。

### §A.6 v2 精修任务状态轮询端点无 owner 校验 -- IDOR
- **位置**: `app.py:4543-4551`
- **严重度**: 中
- **根因诊断**: `/api/ai-refine-v2/status/<task_id>` 只检查 `@login_required`，不校验 `task_id` 是否属于当前用户。`task_id` 是 `pipeline_runner.start_task` 生成的字符串（含时间戳+hex），可预测程度中等。任何已登录用户可枚举 task_id 读取其他用户的精修进度和结果（含产品文案、生成图片 URL）。同理 `/api/single/<task_id>/status`（`app.py:5115-5121`）也缺 owner 校验。
- **修复建议**: 在 `pipeline_runner` 的 task dict 里记录 `user_id`，状态查询时校验 `state["user_id"] == current_user.id`。
- **估时**: 1h
- **依赖**: 需改 `ai_refine_v2/pipeline_runner.py` 的 `start_task` 和 `get_task_status`。

### §A.7 WebSocket legacy 无主批次绕过 owner 校验
- **位置**: `app.py:2234`
- **严重度**: 低
- **根因诊断**: WS 端点 `/ws/batch/<batch_id>` 对 `user_id is None` 的 legacy 批次放行（`batch.user_id is not None and ...`），同类逻辑在列表接口 `app.py:1134`、详情接口 `app.py:1352` 也存在。虽然注释说"阶段六已归档技术债"，但只要 DB 里存在一条 `user_id=NULL` 的批次，任何登录用户都可读取其进度和文件。
- **修复建议**: 一次性 backfill 把所有 `user_id=NULL` 的 Batch 归属到 admin 用户，然后把所有 `user_id.is_(None)` 的 OR 分支删掉，统一为严格 owner 校验。
- **估时**: 1h
- **依赖**: 需先确认 legacy 批次数量和是否仍有业务价值。

### §A.8 Fernet Key 无轮换机制 -- 单点加密密钥泄露即全量明文
- **位置**: `crypto_utils.py:7-17`
- **严重度**: 低
- **根因诊断**: `FERNET_KEY` 是单一全局密钥，模块级缓存 `_fernet_instance`，无 key rotation 策略。一旦该 key 泄露（日志、备份、.env 误提交），所有用户的 `custom_api_key_enc` 可被离线批量解密。项目当前存储的是第三方 AI 平台的 API Key（DeepSeek / ARK），泄露后直接导致用户被薅 quota。
- **修复建议**: (1) 短期：确认 `.env` 不在 git history 中（已有 `.gitignore` 覆盖，需用 `git log --all -- .env` 二次确认）；(2) 中期：引入 key versioning（`FERNET_KEY_V2`），解密时 try v2 → fallback v1，加密一律用最新版。
- **估时**: 1d
- **依赖**: 需要 migration 脚本重新加密所有 `custom_api_key_enc` 字段。
