## 坑点时间线

### 坑 1：Flask context 铁律（一天爆炸两次）

**爆炸一（上午）**：`refine_processor.py` worker 线程内调 `render_template` → 抛 `Working outside of application context`。
**错因**：只包 `with app.app_context():`，但 Flask-Login 的 context processor 在渲染时读 `current_user`，需要完整的 request context。

**爆炸二（晚上）**：`batch_processor.py` 同一 pattern 再次触发。症状更隐蔽 —— 3 个产品 render 失败但 status=done，**前端显示假绿**。
批次 `batch_20260420_019_3932` 实测 3 个产品 render_error 全中。

**修法**：`app.test_request_context()` 替代 `app.app_context()` —— 一次性 push app + request，render_template 链路立刻稳。

**锁死脚本**：
- `scripts/verify_refine_worker_context.py`
- `scripts/verify_render_worker_context.py`

两者都用真 `ThreadPoolExecutor` 跑 worker，确保不变量持续守住。

**详见**：`PROJECT_STATUS_批量生成.md` → "Flask context 铁律" 章节。

### 坑 2：¥10.8 烧钱事故 → 三层防护立碑

**事件**：任务 11 AI 精修首次试跑，用户点"开始精修"后按钮未禁用，在等待 confirm 弹窗期间连点 3 次，后端启动了 3 个并发批次，每批 6 屏 × Seedream，实际扣费 **¥10.8** vs 正常预估 **¥3.6**（3 倍超支）。

**根因**：
- 前端：`#btnStartRefine` 点击后未立刻 `disabled`
- 后端：未做并发启动限制，也无费用硬上限
- 用户心理：弹窗慢 → 以为没响应 → 连点

**立碑 —— 三层防护（缺一不可）**：

1. **前端按钮锁（防误点）**
   - 点击瞬间 `disabled` + loading 态
   - confirm 弹窗期保持锁定
   - fetch 成功/失败都在 `finally` 里解锁

2. **Key 缺失跳转（防空跑）**
   - 后端启动端点第一步查 `ARK_API_KEY`
   - 无 key 直接 302 `/settings`，**不入队**不扣费

3. **硬上限 `MAX_REFINE_COST_PER_RUN`（防意外）**
   - 环境变量默认 ¥50
   - 启动前调 `compute_estimate()` 预估，超额直接 400 拒绝

三层全部落在 commit `e67700b`。配套 mock 测试脚本阻止回归。

**详见**：`PROJECT_STATUS_批量生成.md` → "烧钱 API 必须三层防护" 章节。

## 经验提炼

1. **Worker 线程碰 Flask API = 高危类型**。pattern 级的缺陷必须做 pattern 级的防御（`verify_*.py` 不是装饰，是自动化断言）。

2. **烧钱场景必走三层防护**。防御代码 < 200 行，事故一次 ¥10.8；如果没防护，第二次可能是 ¥100+、¥1000+。

3. **"这次修好就算了" 是错的**。上午修 `refine_processor` 时没做全局 grep，晚上 `batch_processor` 把同一 bug 再演一遍 —— 用 3 个产品的假绿数据换到了教训。

4. **修 bug 的终点是不变量断言，不是"代码能跑"**。`verify_*.py` 锁的是模式不是 bug，未来新增 worker 会被自动检测，不依赖记忆。

5. **PROJECT_STATUS 文档化要实时**。头部元数据过时了 12 天（还写着"任务 4b 待验证"），差点让新对话把昨天的成果当成未完成。写入新技术债的同时一定要同步顶部 TOC。
